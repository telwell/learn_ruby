Performing Code Reviews

Code reviews are a critical part of every single real-world development team's workflow. When you're working at a company and need to merge a new feature or bug fix back into the main repository, often nothing happens until a senior developer or your whole team has had a chance to review your code. The main point is obviously to make sure that bad code doesn't make it into the main repo but it's also a great opportunity to learn.

In an in-person setting, the review might take the form of a full-on meeting in a conference room with a projector, an over-the-shoulder reivew from a colleague, or it might simply be a conversation on Github. For our purposes, we'll combine the conversation on Github with some in-person feedback summarizing our results.

As we've previously mentioned, you'll receive code reviews from instructors and you're highly encouraged to offer unsolicited code reviews to previous students when you read through their code. In this lesson, we'll cover why code reviews are useful and how they will be performed during the course. Read through it now for reference and then refer back to it as necessary later on.

Why Code Reviews are Awesome

As long as you can absorb the critique of your poorly constructed methods, code review is one of the best ways to get better as a developer. It's free feedback from other developers!

On the flip side, performing a code review lets you practice an essential skill as well: reading code. We do a lot of writing code, but you also need to read code in order to get better. It's actually quite difficult to follow other code at first, so developing your "code fluency" is an important step to becoming an effective developer.

How to Code Review (Conceptually)

The process for reviewing code is simple:

  1. Relax and take your time -- you're about to learn something. It's too easy to try and get this out of the way quickly. Do yourself and your partner a favor by putting some thought into it.
  2. Scan the code and understand the project flow. Luckily, you've already done that project yourself so you shouldn't have a problem with this.
  3. Add your comments where appropriate. We'll cover how in the next section.

In a code review, you're looking for three things:

  1. Did your reviewee actually complete the goals of the project or assignment? Refer back to the User Story Acceptance Criteria (if it exists for the project) or the project description.
  2. What did your reviewee do well? You'll always find some "oh cool, I didn't know that!" moment when reviewing. Try to highlight those.
  3. What can your reviewee improve? It's okay to point out style issues (e.g. indentation, variable naming, commenting) in the early weeks because you're all new but try to focus on more semantic improvements in general (e.g. combining overly-verbose lines, improving readability, properly decomposing methods, edge case bugs...).

We understand that you probably feel a bit uncomfortable reviewing code for someone else when you just learned the material yourself -- that's normal and okay. Do the best you can to spot improvements but remember that your suggestions are only one of the benefits of performing code reviews.

How to Code Review (Logistically)

Github is designed for code reviews and conversations!

Because you'll be submitting your assignments and projects in the form of pull requests back to the main project repo, you'll be able to see each of the solutions from your fellow students. All you have to do is find the one for your reviewee and get started.

Pull requests are meant to start a conversation so they offer lots of ways to comment. By clicking into your reviewee's pull request, you'll have options to comment on either the request itself or even individual lines. We'll walk through the steps of the process below.

You'll find the main assignment or project repo under the Viking Education organization on Github (which you should already know since you forked it previously):

How do you actually go through a pull request? The easiest way is to go to the "Files Changed" tab to view the differences. You'll see all the lines that were added or deleted between the pull request and the initial repo. For our assignments, all the files will be new.

You can either comment on a single line by hovering over it in the "Files Changed" tab and clicking the little plus (+) sign that appears...

... Or you can submit a general comment on the repo in the "Conversation" tab:

Regardless of where you comment, that comment will get emailed to the reviewee (and anyone else watching the repo). Any replies to the email automatically get added to the "Conversation" tab, so feel free to start an email back-and-forth or just keep commenting directly in Github's interface.

Once you've submitted your comment(s), you're finished with your code review! Grab a [healthy non-alcoholic beverage] and start on the next lesson or project...
