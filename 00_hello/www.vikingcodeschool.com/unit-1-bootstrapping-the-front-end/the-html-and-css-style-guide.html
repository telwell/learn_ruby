The Viking Markup Style Guide

One of the things that separates novice developers from those with professional experience is a sense of "good style". That means both high level things like organizing markup properly and details like spacing and indentation. Having good style is also important when you're working with other developers because it makes your code much easier to understand and modify.

To help with this, organizations typically produce their own "Style Guides". Style guides codify best practices for how to write and organize your HTML and CSS markup (or other code as well). They get everyone on the same page and writing consistent markup. Well styled markup is not only much more legible to yourself and other developers but it is also better.

In this lesson, we'll lay out the styles we expect you to adhere to when you're working on assignment and projects for Viking Software Consulting. It will give you a good foundation in best practices. Obviously, you're not going to memorize everything here the first time around, but do read through it once and then come back when you're reviewing your projects later.

It's also okay to hack through your markup the first few times. Just make sure you give yourself a bit of time to refactor it once you've finished so you can then apply best practices.

High Level Overview

We want you and your teammates to produce good legible markup, so these are the styles we expect you to adhere to. Code reviews are good times to bring up style violations (though focus on the larger issues first).

On a high level, the styles below represent the following principles:

  1. Be well organized so your markup is more legible and maintainable
  2. Try to re-use code where you can and be flexible for that re-use
  3. Whatever you do, be consistent

You should come back to this guide if you have questions about style. If something isn't in the guide, ask us and we'll clarify (and add it).

HTML Styles

One ID, Many Classes

Use classes when there are (potentially) multiple versions of an item and IDs only when it is completely unique.

Good candidates for ids: header, footer, modal popups.

Bad candidates for ids: navigation, item listings, item view pages

Use Semantic Class and ID Naming

Use semantic class names wherever possible. That means that you shouldn't name something based on how it's supposed to be styled but rather based on what it is. That gives you much more flexibility later to change around the styling.

Bad: <p class="red">

Good: <p class="alert">

Use Proper Elements

There's a semantic tag for just about everything. Headings should be in <h1>...<h6> tags and text should be wrapped in <p> tags. Use the HTML5 sectioning elements like <section> and <article> where they are appropriate.

Bad:

<div class="header">
  Welcome to Happy Puppy Fanpage! 
  The only site with the happiest puppies
</div>
This is the best puppy fanpage on the interwebs, 
and we promise to show you nothing but the 
happiest puppies there are.  This week, we'll 
be featuring a corgi by the name of ...<br><br>
<a href="mailto:contact@happypuppyfanpage.com">
  Contact us</a> for more information.

Good:

<main>
  <header>
    <h1>
      Welcome to Happy Puppy Fanpage!
    </h1>
    <h2>
      The only site with the happiest puppies
    </h2>
  </header>
  <p>
    This is the best puppy fanpage on the interwebs, 
    and we promise to show you nothing but the 
    happiest puppies there are.
  </p>
  <p>
    This week, we'll be featuring a corgi by the name of ...
  </p>
</main>
<footer>
  <a href="mailto:contact@happypuppyfanpage.com">Contact us</a>
   for more information.
</footer>

Alt Attributes for Images

Use alt text on images and be descriptive about what the image actually is. This is really important for screen readers and SEO alike.

Bad: <img src="puppies.jpg">

Good: <img src="puppies.jpg" alt="cute puppies playing in the grass">

Element Namespacing (ie. "Encapsulation")

"Namespacing" is really just organizing logical groups of things under a specific name (ie. encapsulating them). For instance, if all of your products are listed on the page, all the styles within that grouping might be namespaced under "product". This is useful because it makes sure that your styles are narrowly targeted at just the elements inside that namespace.

You can do this a couple of ways -- you might prepend a specific string like "products" to all class names, e.g. <ul class="products-listing">, though this can get a bit wonky if the file is too large.

You can also nest elements within each other, for instance wrapping all products in a <div class="products"> and then target styles in your stylesheet with

.products .listing{ 
  color: red; 
}
.products .feature-box{
  padding: 20px 0;
}

This is easy with a preprocessor like SASS, where you can easily nest these styles and include any other relevant ones within that hierarchy:

.products{
  .listing{
    color: red
  }
  .feature-box{
    padding: 20px 0;
  }
  /* Other product-related styles here */
}

If you fail to namespace properly, different items that are named the same thing might have unpredictable behavior, for instance two different "alert" classes on separate pages.

CSS Styles

Properly Indent and Space

2 spaces is conventional. Don't use tabs (or at least make sure they convert to spaces) because different programs treat them differently.

Bad:

button,.alert-box{ color: red; font-size: 15px; }

Good:

button,
.alert-box{
  color: red;
  font-size: 15px;
}

Proper Naming

Use lowercase names with hyphen delimiters.

Bad: <div class="alertBox">

Good: <div class="alert-box">

Use Comments and Line Spacings for Sections

CSS files will quickly become quite large, even if you're separating them out wherever possible. Use at least 4 blank lines between major sections of code and write comments atop each major section describing what it is meant to do.

When using SASS, as below, it's doubly important to be conscious of your spacing and commenting.

Good:

.othersection{
  margin: 20px 20px 20px 60px;
  font-size: 1.5rem;
  line-height: 1.35em;
}



/* 
 * Functionality of the FAQ menu accordian
 * on the main page 
 */
.faq{
  margin-top: 40px;
  .question{
    margin: 1px;
    color: #eee;
    font-weight: 100;
    letter-spacing: 1px;
  }
  ...
}

Choose Classes over IDs

When in doubt, use a class. IDs are very specific and, while you may not intend to duplicate the element in HTML later, you might want to use its styles on something else as well.

Spend some time having to convert big chunks of IDs to classes and you'll see what we mean.

Break Up Your CSS

Break chunks of CSS into different files if necessary. If you're using a build tool that aggregates your CSS for you (like Rails does), this is free functionality so use it!

If you aren't using a build tool, you'll have to weigh the time cost of performing additional HTTP requests to fetch each new CSS file with the productivity benefits of maintaining good modular files. While you're learning, it is better to just break out unrelated style modules into multiple CSS files and include them each individually. For a production project, you will certainly use a build tool to aggregate them anyway.

Don't Rely on Order

If all other specificity factors are the same, the last CSS style used takes precedence. This is extremely brittle because you might easily forget later that order matters and rearrange your CSS, breaking your layout.

Bad:

.alert-box{
    color:red;
}
.alert-box{
    color:blue;
}

...this will produce a blue alert because that's the last style. When you will be tempted to do this, it won't be nearly so clear cut. Don't do it!

Don't Rely on !important

If you need to use !important to override other styles, for instance:

a.special-link{
  color: red!important;
}

...there are probably larger issues in your organization that need to be addressed. !important will take priority over all other specificities but its power must be used sparingly.

Stick to the Right Specificity

Many resources urge you to avoid too much specificity in your selectors. This is related to namespacing... for styles that need to be re-used (like forms), you shouldn't bury them too deeply within other classes or IDs. But if they don't need to be re-used, give them enough specificity so they don't pollute the namespace of other totally unrelated modules.

The rule of thumb here is "Make your selectors just specific enough that they won't bleed into other areas". It takes some practice.

Don't Make CSS (and SASS) Mimic HTML Structure

Another rule of thumb is that you shouldn't make your CSS mimic your HTML structure. That can be very tempting when you're nesting with SASS. The problem is that, when your SASS styles are so deeply nested, any high level change to your HTML page struture will blow up all your downstream styles as well.

Another way this has been phrased is the "Inception Rule" (from the movie), or "Don't nest more than 4 levels deep".

Use Shorthand Properties and Values

Properties like margin and padding have the shorthand syntax which combines the margin-top, margin-bottom etc. into one. Use these shorthand properties unless you're only trying to target a single specific property like padding-left.

Bad:

button{
  padding-top: 5px;
  padding-left: 10px;
  padding-bottom: 2px;
  padding-right: 10px;
}

Good:

button{
  /* Note that the last value would be redundant */
  padding: 5px 10px 2px;  
}

Use Shorthand Hex Values

Any hex value with repeated digits can be cut in half.

Bad: #44ffaa

Good: #4fa

Drop the Units from Zeros

Bad: margin-top: 0px;

Good: margin-top: 0;

Align Vendor Prefixes by Value

The vendor prefixes (which guarantee that certain styles apply in certian browsers) are somewhat annoying but frequently used for compatability. When writing them, align the spacing on the values.

Bad:

.panel{
  -webkit-box-sizing: border-box;
  -moz-box-sizing: border-box;
  box-sizing: border-box;
}

Good:

.panel{
  -webkit-box-sizing: border-box;
     -moz-box-sizing: border-box;
          box-sizing: border-box;
}

An Example Style Guide

Browse through the Github Style Guide for examples of how they are used in the real world. Don't worry about absorbing everything though.

Wrapping Up

In the coming assignments and projects, you'll get plenty of experience applying these things. You'll also be a part of numerous code reviews where you'll see your teammates' styles and get feedback on your own. It will very quickly become natural and you'll soon be writing professional-looking markup.
